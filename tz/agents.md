# @tz — Папка технических заданий

## Назначение папки

`tz/` (от «техзадание») — это **бэклог спецификаций и отчётов**. Здесь живут markdown-файлы, которые описывают задачи, которые нужно выполнить в проекте, или фиксируют состояние работ (отчёты об ошибках, планы, и т.п.).

Файлы из `tz/` **не выполняются автоматически**. Это документы для чтения человеком или ИИ-агентом. Чтобы запустить работу по ТЗ, нужно явно попросить агента: «возьми `tz/<файл>.md` и сделай».

## Что лежит в `tz/`

| Тип файла | Назначение | Примеры |
|-----------|-----------|---------|
| **TZ-цель** (`*-plan.md`, `update-*.md`) | Декомпозиция большой задачи на подзадачи с границами ответственности | `cleanup-plan.md`, `update-design-tests.md` |
| **TZ-обзор** (`<topic>.md`) | Верхнеуровневое описание цели, без детального плана | `cleanup.md`, `structure.md` |
| **Отчёт** (`*-errors.md`, `*-fails.md`) | Сводка реально обнаруженных проблем, с группировкой и привязкой к коду | `test-errors.md`, `test-fails.md` |

Если файла нужного типа нет — не выдумывай, посмотри существующие и следуй их стилю.

## Соглашения по именованию

| Паттерн | Когда использовать |
|---------|-------------------|
| `<topic>.md` | Краткое ТЗ без плана (одна страница описания) |
| `<topic>-plan.md` | Развёрнутый план с разбивкой по агентам/шагам |
| `<topic>-errors.md` | Отчёт об ошибках (e2e, сборка, lint) |
| `<topic>-fails.md` | Отчёт о провалах (тесты, проверки) |
| `update-<что>-<зачем>.md` | ТЗ на изменение существующей функциональности |

Имена в нижнем регистре, через дефис, без пробелов и спецсимволов. Префикс даты — **не** в имени файла (даты живут в заголовке).

## Структура ТЗ-файла

Качественное ТЗ содержит:

1. **Заголовок H1** — суть задачи в одном предложении.
2. **Контекст / Why** — зачем это нужно, какая боль закрывается.
3. **Цель / Goal** — измеримый результат (что должно работать/быть по итогу).
4. **Задачи / Steps** — конкретные шаги с указанием файлов и областей.
5. **Границы ответственности** (если работает несколько агентов) — кто что делает, кого не трогает.
6. **Что НЕ делать** — анти-паттерны и запрещённые изменения.

Если ТЗ ссылается на код — обязательны конкретные пути и имена файлов. «Поправь авторизацию» — плохое ТЗ. «В `apps/backend/src/auth/auth.service.ts` заменить bcrypt на argon2» — хорошее.

## Как работать с ТЗ

### 1. Прочитать полностью

Перед стартом прочитай файл целиком (через `read` с `offset/limit`, если большой). Не действуй по первому абзацу — часто важные ограничения в конце.

### 2. Сверить с реальным кодом

ТЗ пишется «о мире», а код — «как есть». Перед любыми правками:

- Проверь, что упомянутые файлы/модули существуют (`bash: find`, `ls`).
- Если файл/модуль уже удалён или переименован — **не делай вид, что работаешь по старому**, обнови ТЗ.
- Если в ТЗ несколько взаимоисключающих пунктов — уточни у пользователя, не выбирай сам.

### 3. Выбрать режим работы

| Режим | Когда |
|-------|-------|
| **Сам через `read`/`edit`/`write`** | ТЗ маленькое, изменения точечные, не нужна изоляция контекста |
| **Субагент `subagent({ agent: ... })`** | Большое ТЗ, есть декомпозиция в плане, нужна изоляция |
| **Цепочка `chain: [...]`** | Шаги зависят друг от друга, передают результат дальше |
| **Параллель `tasks: [...]`** | Независимые куски (например, разные области: backend, frontend, docs) |

### 4. Не раздувать контекст

Если в плане уже есть разбивка по агентам (`cleanup-plan.md` хороший пример) — **запускай субагентов**, а не делай всё сам. Главный агент координирует, субагенты исполняют.

### 5. Не додумывать

Если в ТЗ не сказано «обновить README» — не обновляй. Если нужно выйти за рамки — сначала спроси пользователя.

## Как добавить новое ТЗ

1. **Выбери тип**: цель / план / отчёт.
2. **Используй шаблон** из раздела «Структура ТЗ-файла».
3. **Имена файлов** — по таблице выше.
4. **Не дублируй** существующие ТЗ: если задача перекликается — добавь ссылку, а не новый файл.
5. **Ссылайся на реальные пути** в репозитории (`apps/backend/src/...`, `e2e/tests/...`), а не абстрактные.

## Шаблон нового ТЗ (копируй при необходимости)

```markdown
# <Глагол> <что> <зачем>

## Контекст

<Почему задача возникла, на что влияет, какая боль>

## Цель

<Что должно получиться на выходе, как измерить успех>

## Задачи

1. **<Область 1>** — <что сделать, какие файлы>
2. **<Область 2>** — ...

## Границы ответственности (если несколько агентов)

- **agent-a**: <что делает>, не трогает <что>
- **agent-b**: <что делает>, не трогает <что>

## Что НЕ делать

- <явный запрет 1>
- <явный запрет 2>
```

## Связь с субагентами проекта

Качественное ТЗ позволяет делегировать работу **субагентам из `.pi/agents/`**:

| Субагент | Когда подходит для работы по ТЗ |
|----------|-------------------------------|
| `html-designer` | ТЗ про вёрстку / UI / Tailwind / дизайн-систему |
| `browser-checker` | После реализации — визуальная проверка результата в браузере |
| `test-runner` | Запустить e2e, описанные в ТЗ, и вернуть отчёт |
| `test-fixer` | Если в ТЗ указано «почини упавшие тесты» |
| `docs-maintainer` | Если ТЗ меняет код, и нужна актуализация `docs/` |

Пример: «возьми `tz/test-fails.md`, почини через `test-fixer`, потом через `docs-maintainer` обнови `docs/`».

## Частые ошибки

- ❌ **Писать ТЗ с кодом внутри, а не ссылками** — устаревает за неделю.
- ❌ **Копировать ТЗ из старого проекта без ревизии** — половина пунктов нерелевантна.
- ❌ **Игнорировать раздел «Что НЕ делать»** — приводит к расползанию скоупа.
- ❌ **Запускать субагента по ТЗ-цели без `-plan.md`** — субагент не знает границ и делает «всё».
- ❌ **Оставлять «висячие» ТЗ** — если задача выполнена, перенеси файл в `tz/done/` или удали (см. ниже).

## Жизненный цикл ТЗ

```
создано ──► взято в работу ──► выполнено ──► done/   (или удалено, если одноразовое)
                │
                └─► отменено ──► tldr/   (с пометкой «отменено, причина»)
```

- **`tz/done/`** — выполненные ТЗ, оставлены как история решений. Можно подглядывать при оценке скоупа новых задач.
- **`tz/tldr/`** — отменённые/устаревшие ТЗ с короткой пометкой почему.

Перенос в `done/` — **явное действие**: после успешного завершения работы по ТЗ агент (или пользователь) перемещает файл сам или просит об этом.

## Быстрый чек-лист перед стартом работы по ТЗ

- [ ] Файл прочитан целиком
- [ ] Упомянутые пути в репозитории существуют
- [ ] Понятен результат (что должно работать/быть)
- [ ] Понятны границы (что НЕ трогать)
- [ ] Выбран режим: сам / субагент / цепочка
- [ ] Если субагент — его `agent.md` совместим с задачей
- [ ] Результат можно проверить (тест / сборка / ручная проверка через browser-checker)
