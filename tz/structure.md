Проанализируй текущий проект и создай структуру контекста для команды субагентов.

Шаг 1. Изучи проект:
- Прочитай package.json, README, docker-compose.yml, nginx.conf, .env.example (если есть)
- Пройди по src/ (или app/, lib/, backend/, frontend/) и определи ключевые области
- Найди конфиги тестов (jest.config, playwright.config, vitest.config и т.д.)
- Найди CI/CD файлы (.github/workflows/, Dockerfile)

Шаг 2. Создай директорию docs/ в корне проекта и наполни её файлами:

docs/
├── context-index.md          — карта всех контекстов
├── project-overview.md       — архитектура, стек, entry points
├── areas/[area]-structure.md       — для каждой логической области (auth, api, db, ui, payments и т.д.)
├── testing-guide.md          — как запускать тесты, что проверять, известные проблемы
├── deployment.md             — Docker, nginx, env-переменные, порты
└── agents.md                 — сводка для субагентов (см. Шаг 3)

Каждый [area]-structure.md должен содержать:
- Назначение области (2-3 предложения)
- Ключевые файлы и их роль (список)
- Входные/выходные данные (API endpoints, props, events)
- Зависимости от других областей
- Частые ошибки или неочевидные моменты
- Что проверять при изменениях (чек-лист для тестировщика)

Лимит: каждый файл не более 300 строк. Если область большая — разбей на подобласти.

Шаг 3. Создай docs/context-index.md:
# Context Index

| Файл | Область | Когда использовать |
|------|---------|-------------------|
| project-overview.md | Общая архитектура | Новому агенту для понимания стека |
| auth-structure.md | Аутентификация | Тестирование login/logout, сессии |
| ... | ... | ... |
Требования:
Не используй субагентов для этой задачи — делай сам через read_file/write_file
Если не уверен в назначении файла — прочитай его (не гадай по имени)
Не включай в контекст чувствительные данные: пароли, токены, ключи API (замени на [REDACTED])
